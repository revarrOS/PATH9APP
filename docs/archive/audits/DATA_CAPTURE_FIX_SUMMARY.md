# Data Capture Fix Summary
**Date**: December 29, 2025
**Status**: CRITICAL P0 BLOCKER FIXED

---

## Problem Identified

**Initial Assessment**: Service router and orchestrate existed but weren't saving data to database.

**Root Cause**: Services were configured to NOT save data:
1. `translate-medical` was called with `save_to_diagnosis: false`
2. `translate-medical` required `diagnosis_metadata` but wasn't receiving it
3. Services expected user auth tokens but orchestrate was calling with service role key

---

## Changes Made

### 1. Service Router - Enable Data Saving

**File**: `supabase/functions/orchestrate/service-router.ts`

**Changes**:
- Line 100: Changed `save_to_diagnosis: false` → `save_to_diagnosis: true`
- Lines 63-85: Added diagnosis parsing logic to extract:
  - Diagnosis name (CLL, AML, or from "diagnosed with X" pattern)
  - Diagnosis date (defaults to today)
  - Stage/severity (from "stage X" pattern)
- Lines 101-106: Added `diagnosis_metadata` object with parsed values
- Line 94: Added `user_id: userId` to request body
- Line 134: Added `user_id: userId` to understand-appointment call

**Impact**: Diagnosis information is now extracted and passed to translate-medical service with save flag enabled.

---

### 2. Translate-Medical Service - Accept Service-to-Service Calls

**File**: `supabase/functions/translate-medical/index.ts`

**Changes**:
- Line 13: Added `user_id?: string` to TranslateMedicalRequest interface
- Lines 42-67: Restructured auth logic:
  - Parse body FIRST (to check for user_id)
  - If `body.user_id` exists, trust it (service-to-service call)
  - Otherwise, validate auth token (direct user call)
- Removed duplicate JSON parsing code

**Impact**: Service can now be called from orchestrate with service role key + user_id in body.

---

### 3. Understand-Appointment Service - Accept Service-to-Service Calls

**File**: `supabase/functions/understand-appointment/index.ts`

**Changes**:
- Line 13: Added `user_id?: string` to UnderstandAppointmentRequest interface
- Lines 36-61: Restructured auth logic (same pattern as translate-medical):
  - Parse body FIRST
  - If `body.user_id` exists, trust it
  - Otherwise, validate auth token

**Impact**: Service can now be called from orchestrate with service role key + user_id in body.

## How It Works Now

### User Flow: "I was diagnosed with CLL"

**Step 1**: User sends message to Today tab
```typescript
{
  user_message: "I was diagnosed with CLL",
  user_id: "uuid",
  journey_state: {...},
  ...
}
```

**Step 2**: Orchestrate receives message and classifies intent
```typescript
intent: {
  primary_intent: "medical_translation",
  requires_medical_translation: true,
  confidence: 0.9
}
```

**Step 3**: Service router extracts diagnosis info
```typescript
{
  diagnosisName: "Chronic Lymphocytic Leukemia (CLL)",
  diagnosisDate: "2025-12-29",
  stage: undefined
}
```

**Step 4**: Service router calls translate-medical
```typescript
POST /functions/v1/translate-medical
{
  user_id: "uuid",
  technical_text: "I was diagnosed with CLL",
  save_to_diagnosis: true,
  diagnosis_metadata: {
    diagnosis_name: "Chronic Lymphocytic Leukemia (CLL)",
    diagnosis_date: "2025-12-29"
  }
}
```

**Step 5**: Translate-medical service:
- Trusts user_id from body (service-to-service call)
- Calls LLM to create plain English explanation
- Saves to diagnoses table (lines 111-125):
```sql
INSERT INTO diagnoses (
  user_id,
  diagnosis_name,
  diagnosis_date,
  raw_pathology_text,
  plain_english_summary
) VALUES (...)
```

**Step 6**: User sees response in Today tab
**Step 7**: User navigates to My Path tab
**Step 8**: My Path tab loads diagnosis from database
**Step 9**: DiagnosisExplainer component displays:
- Diagnosis name: "Chronic Lymphocytic Leukemia (CLL)"
- Diagnosis date: "December 29, 2025"
- Plain English summary: (generated by LLM)

## User Flow: "I have an appointment with Dr. Smith on January 15"

**Step 1-2**: Same as diagnosis flow, but intent is:
```typescript
intent: {
  primary_intent: "appointment_mention",
  requires_appointment_understanding: true,
  confidence: 0.85
}
```

**Step 3**: Service router calls understand-appointment
```typescript
POST /functions/v1/understand-appointment
{
  user_id: "uuid",
  user_message: "I have an appointment with Dr. Smith on January 15",
  save_to_database: true
}
```

**Step 4**: Understand-appointment service:
- Trusts user_id from body
- Parses appointment details using AppointmentParser
- If confidence >= 5 and appointment_datetime exists:
  - Saves to appointments table
  - Saves provider to care_team table

**Step 5**: User navigates to My Path tab
**Step 6**: AppointmentCard component displays:
- Date: January 15, 2025
- Provider: Dr. Smith
- Role: (extracted from parser)
- Role explanation: (e.g., "Your oncologist coordinates your cancer treatment...")
